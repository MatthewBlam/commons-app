## 2. Security Issues

### 2.1 CRITICAL — `openExternal` has no URL validation

**File:** `src/main/ipc/handlers.ts:57-59`

```typescript
ipcMain.handle("app:open-external", async (_, url: string) => {
  await shell.openExternal(url);
});
```

`shell.openExternal()` with an unvalidated URL is a well-known Electron
vulnerability ([Electron security docs](https://www.electronjs.org/docs/latest/tutorial/security#15-do-not-use-shellopenexternal-with-untrusted-content)).
A compromised renderer or crafted input could pass `file://`, `javascript:`, or
OS-specific protocol handlers that execute arbitrary commands.

**Fix:** Validate that the URL scheme is `http` or `https` before opening.

### 2.2 HIGH — No IPC input validation on secret key names

**File:** `src/main/ipc/handlers.ts:9-15`

The `secrets:save` and `secrets:load` handlers accept any string as a `key`. If
the renderer process is ever compromised, an attacker could overwrite any setting
in the database (including `embedding_provider`, which controls application behavior)
or read any encrypted value.

**Fix:** Validate `key` against an allowlist of expected secret names
(`cohere_api_key`, `notion_token`, `google_tokens`).

### 2.3 MEDIUM — Secrets and settings share the same table

**File:** `src/main/db/migrations.ts`, `src/main/auth/storage.ts`

Encrypted secrets (API keys, tokens) and plain-text settings
(`embedding_provider`) coexist in the `settings` table. This means:

- A setting read accidentally returns encrypted binary data
- No clear access boundary between sensitive and non-sensitive values
- Database inspection reveals which secret keys exist

Not blocking for MVP but worth separating before shipping.

### 2.4 LOW — Cohere key validation uses raw HTTP instead of SDK

**File:** `src/main/ipc/handlers.ts:17-36`

The plan specifies using `CohereClientV2` from `cohere-ai` SDK. The implementation
uses raw `fetch` with manually constructed headers. The SDK would handle retries,
error parsing, and TLS validation. The `cohere-ai` package isn't even installed yet.

## 3. Architectural Issues

### 3.1 Root `tsconfig.json` still has deprecated `baseUrl`

**File:** `tsconfig.json`

```json
"compilerOptions": {
  "baseUrl": ".",
  "paths": { "@renderer/*": ["src/renderer/src/*"] }
}
```

`tsconfig.web.json` was fixed to remove `baseUrl`, but the root `tsconfig.json`
still has it. This will emit a deprecation warning in TS 6 and break in TS 7.
The root tsconfig's `paths` should use `"./src/renderer/src/*"` and drop `baseUrl`.

### 3.2 `ChunkRow.embedding` type mismatch between layers

**File:** `src/main/db/database.ts:127` vs `src/shared/types.ts:29`

- `shared/types.ts` defines `Chunk.embedding: Uint8Array | null` (fixed in this session)
- `database.ts` defines `ChunkRow.embedding: Buffer | null` and `ChunkDbRow.embedding: Buffer | null`

`Buffer extends Uint8Array` so assignment from DB → shared type works, but going
the other direction (renderer creates a `Uint8Array`, passes it to main process)
would cause type errors. These should be consistent. Since the database layer runs
in Node.js and `better-sqlite3` returns `Buffer` for BLOBs, `Buffer` in the DB
layer is correct, but the mapping function should convert to `Uint8Array` when
producing the shared `Chunk` type.

### 3.3 Packaging tool mismatch — `electron-builder` vs `electron-forge`

**Files:** `electron-builder.yml`, `package.json`

The plan (Task 12) specifies `electron-forge` with `maker-dmg` and `maker-squirrel`.
The codebase uses `electron-builder` with its own YAML config. Both are valid
choices, but:

- The plan should be updated to match the actual tool, OR
- `electron-builder` should be replaced with `electron-forge`

Mixing or switching later creates unnecessary churn.

### 3.4 Synchronous `safeStorage` instead of async

**File:** `src/main/auth/storage.ts`

The plan specifies async methods (`encryptStringAsync` / `decryptStringAsync`).
The implementation uses synchronous `encryptString` / `decryptString`, which block
the main process during encryption. For small API keys this is negligible, but
it's a bad pattern that will scale poorly if tokens get larger (Google OAuth token
JSON objects, etc.).

### 3.5 Database not closed on app shutdown

**File:** `src/main/db/singleton.ts`, `src/main/index.ts`

`closeDb()` exists but is never called. Without it:

- WAL checkpoint may not complete, risking data loss on crash
- SQLite file lock may not release cleanly on quit

**Fix:** Add to `src/main/index.ts`:

```typescript
app.on("will-quit", () => {
  closeDb();
});
```

### 3.6 `input.tsx` exports non-component `InputPrimitive`

**File:** `src/renderer/src/components/ui/input.tsx:43`

```typescript
export { InputPrimitive };
```

Same React Fast Refresh issue that was just fixed for `buttonVariants` in
`button.tsx`. Files that export non-component values alongside components break
hot reloading.

## 4. Build & Test Issues

### 4.1 CRITICAL — All tests fail due to native module version mismatch

```
The module 'better_sqlite3.node' was compiled against NODE_MODULE_VERSION 137.
This version of Node.js requires NODE_MODULE_VERSION 127.
```

**Root cause:** `postinstall` runs `electron-rebuild`, which compiles
`better-sqlite3` for Electron's Node.js (v22 → ABI 137). Vitest runs under
system Node.js v22.23.1 (ABI 127). The compiled native module is incompatible.

The `test` script tries `pnpm rebuild better-sqlite3` before running Vitest, but
this rebuilds for the system Node which then gets overwritten by the next
`pnpm install`. This is a circular conflict.

**Fix:** Either:

1. Use `vitest` with Electron's Node via `electron-vite test` or `xvfb-run electron ./node_modules/.bin/vitest`
2. Configure separate build targets, e.g. `better-sqlite3-builds` for both ABI versions
3. Remove `postinstall: "electron-rebuild"` and run it only before `dev`/`build` commands, leaving the default build for tests

### 4.2 Dozens of Prettier formatting violations

All files in `src/main/` and `src/preload/` have line-length violations. Running
`pnpm format` would fix them all, but the code was clearly written without running
the formatter. CI should enforce formatting to prevent this.

### 4.3 `electron-builder.yml` has scaffold defaults

| Field                | Current                            | Should be           |
| -------------------- | ---------------------------------- | ------------------- |
| `appId`              | `com.electron.app`                 | `com.commons.app`   |
| `productName`        | `commons-app-scaffold`             | `Commons`           |
| `win.executableName` | `commons-app-scaffold`             | `commons`           |
| `publish.url`        | `https://example.com/auto-updates` | real URL or removed |

The main process sets `com.commons.app` via `electronApp.setAppUserModelId()`,
creating a mismatch with the builder config.

### 4.4 `pnpm-workspace.yaml` has unresolved placeholder

```yaml
sharp: set this to true or false
```

This will cause pnpm to treat the string as the value, not a boolean. Either
set it to `true`/`false` or remove the line.

## 5. Consistency & Best Practice Issues

### 5.1 Linux targets exist despite plan excluding Linux

**Files:** `electron-builder.yml` (AppImage/snap/deb targets), `src/main/index.ts:14`
(Linux icon conditional)

The plan says "macOS (DMG) + Windows (Squirrel) only. No Linux for MVP." Remove
Linux targets to avoid confusion about what's tested and supported.

### 5.2 `react` and `react-dom` in `devDependencies`

**File:** `package.json`

React is listed under `devDependencies`. While this works (Vite bundles everything),
it's unconventional. Libraries that are part of the runtime output belong in
`dependencies`. Some tooling (e.g., `depcheck`, Renovate) may flag this.

### 5.3 OllamaOption effect lacks cleanup

**File:** `src/renderer/src/components/setup/OllamaOption.tsx:12-15`

```tsx
useEffect(() => {
  checkOllama();
}, []);
```

Unlike the fixed `App.tsx` effect which uses an `ignore` flag, this effect doesn't
guard against state updates after unmount. If the user navigates away before the
check completes, React will warn about setting state on an unmounted component.

### 5.4 Missing `eslint-disable` or `checkOllama` dep in OllamaOption

The `useEffect` references `checkOllama` which is defined outside the effect but
closes over `setStatus`/`setModels`. The `react-hooks/exhaustive-deps` rule would
flag the missing dependency (if not suppressed by the `eslint-plugin-react-hooks`
version in use).

### 5.5 No `.node-version` or `.nvmrc` file

The plan requires Node.js 20.19+ or 22.12+. There's no version pinning file, so
collaborators could run any Node version and encounter cryptic native module errors.

### 5.6 `"use client"` directives in Electron app

**Files:** `button.tsx:1`, `card.tsx:1`, `input.tsx:1`

These come from the Coss UI templates (Next.js-oriented). They're harmless in
Electron but misleading — there's no server rendering here. Consider removing
them to avoid confusion.

## 6. What's Working Well

These are the things that are correctly implemented and align with the plan:

- **Electron security model:** `contextIsolation: true`, `nodeIntegration: false`,
  `sandbox: true` — the gold standard for Electron security
- **Content Security Policy:** Properly scoped to `self`, Cohere API, and Ollama localhost
- **SQLite configuration:** WAL mode and foreign keys enabled via pragmas
- **Migration system:** Clean versioned migrations with idempotency, wrapped in transactions
- **Cascade deletes:** Properly configured in schema and verified in tests
- **Typed IPC bridge:** Preload exposes a typed `CommonsAPI` via `contextBridge`
  with matching `index.d.ts` declaration
- **Database query helpers:** Well-structured with proper snake_case ↔ camelCase
  mapping between DB rows and TypeScript interfaces
- **UI component system:** Coss UI (Base UI variant) correctly set up with Tailwind v4
  theme variables, dark mode support, and component composition via `useRender`
- **Setup flow:** Clean UX with Cohere/Ollama choice, validation, back navigation
- **Test coverage intent:** Database tests cover CRUD, cascades, idempotency, and
  settings — good coverage for the foundation layer
