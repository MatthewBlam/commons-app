# Commons — Production Readiness Audit

**Date:** 2026-07-13
**Scope:** Full codebase — main process, preload, renderer, build/packaging, dependencies
**Verdict:** **Not ready to ship.** Three issues are release blockers on their own, and one of them silently and permanently destroys user data.

**Verification performed:** `pnpm typecheck` (clean) · `pnpm test` (151/151 pass) · `pnpm lint` (**21 errors**) · `pnpm audit --prod` (clean) · production build + bundle string inspection · two purpose-written failing tests proving C2 and H4.

---

## Executive summary

The security fundamentals are genuinely good, and I want to be clear about that up front: context isolation, sandbox, CSP, `safeStorage`, PKCE with state validation, parameterized SQL everywhere, an allowlist on IPC secret keys, and zero XSS surface. That work is done, and done properly. The hard architectural calls — hybrid FTS+vector retrieval with RRF, incremental sync via content hashing, local-first storage — are all sound.

What stands between this and a release is a different category of problem:

1. **You cannot ship the app as configured.** It is ad-hoc signed, so macOS Gatekeeper will refuse to open it for anyone who downloads it.
2. **Your OAuth client secrets are printed in the shipped binary.** I extracted them from `dist/main/index.js`.
3. **Telemetry is on by default and there is no way to turn it off.** The opt-out IPC exists; no UI ever calls it.
4. **A transient network blip during sync permanently removes documents from search** — silently, unrecoverably.
5. **There is no auto-updater**, so every bug you ship is permanent for that user.

The recurring theme across the rest: **error paths were written but never exercised.** The database corruption recovery doesn't work, the migration backup is incomplete, and a failed sync flashes its error for 1.5 seconds and then tells the user "Done."

## CRITICAL

### C1 — OAuth client secrets ship in plaintext inside the app bundle

`electron.vite.config.ts:17-23` uses Vite `define` to inline env vars into the main bundle. `define` is literal text substitution, so the values are baked into the shipped JavaScript. Verified against a real build:

```
LEAKED  NOTION_CLIENT_ID      present verbatim in dist/main/index.js
LEAKED  NOTION_CLIENT_SECRET  present verbatim in dist/main/index.js
LEAKED  GOOGLE_CLIENT_ID      present verbatim in dist/main/index.js
LEAKED  GOOGLE_CLIENT_SECRET  present verbatim in dist/main/index.js
LEAKED  POSTHOG_API_KEY       present verbatim in dist/main/index.js
```

Anyone who downloads Commons can unpack the asar and run `strings` on it. **`NOTION_CLIENT_SECRET` is the serious one** — Notion treats it as a confidential client secret with no public-client mode. With it, an attacker can stand up a convincing OAuth consent screen branded as _Commons_, exchange authorization codes for real Notion tokens, and read a victim's entire workspace. They can also burn your quota or get your integration suspended.

Google's _Desktop app_ client type explicitly treats its secret as non-confidential, so that half is lower-risk — but it still shouldn't be published. `POSTHOG_API_KEY` is a write-only project key designed to be public; that one is fine.

**Fix:**

- **Rotate the Notion and Google client secrets now.** They are in your `.env` and in every build you have produced.
- Move token exchange behind a minimal backend you control (a Cloudflare Worker is enough): the app does PKCE and posts the code to your endpoint, the endpoint holds the secret and returns the token. This is the only correct pattern for Notion.
- For Google, alternatively register a **Desktop app** OAuth client and rely on PKCE alone.

### C2 — A failed embedding permanently removes a document from search

**This is the one that loses user data, and it is completely silent.**

In `src/main/sync/sync-manager.ts`, `insertDocument` (line 120) persists the new `content_hash` **before** the embedding call (line 160) has succeeded. When embedding throws — a Cohere 429, a rate limit, a dropped connection — the `catch` at line 191 marks the document `error`, but the new hash is already committed and the document has **zero chunks**.

On the next sync, the skip check at lines 112-117 sees a matching `content_hash`, finds zero chunks, computes `modelMatches = true` (because `chunks.length === 0`), and **returns early**. The document is never re-embedded. It is invisible to search forever. No amount of re-syncing fixes it — only editing the document at the source (changing its hash) or "Clear all data" recovers it.

I wrote a test to confirm. Sync once with embedding failing, then again with embedding healthy:

```
AssertionError: expected "vi.fn()" to be called 2 times, but got 1 times
 ❯ expect(embedDocuments).toHaveBeenCalledTimes(2)
```

The second sync never even _attempted_ to embed. Final state: `sync_status='error'`, a valid `content_hash`, and zero chunks — permanently.

`resetStalePendingDocuments` (`src/main/index.ts:100`) compounds this: a crash or quit mid-sync flips every in-flight document to `error` with hashes already written. An interrupted first sync can strand a large fraction of the corpus.

**Fix:** persist `content_hash` only _after_ chunks are successfully stored — write it inside `replaceChunksForDocument`'s transaction. Or, minimally, make the skip check at line 112 also require `existing.syncStatus === 'synced'`. Add the regression test; this is exactly what tests are for.

### C3 — Telemetry is on by default and the user cannot turn it off

`src/main/index.ts:109` fires `commons_app_opened` to PostHog on the very first launch — before the user has seen a single screen or agreed to anything. The default is opt-out (`posthog.ts:22`).

`getTelemetryEnabled` / `setTelemetryEnabled` are wired through the preload (`src/preload/index.ts:85-88`) and the IPC handlers exist (`handlers.ts:325-331`) — but `grep -rni telemetry src/renderer/` returns **zero hits**. The opt-out was plumbed all the way to the context bridge and then never given a switch. **There is no way for a user to disable analytics.**

There is also no `PRIVACY.md`, no `LICENSE`, and no `README` in the repo.

For an app that indexes a student club's private documents, shipping silent, default-on, un-disableable analytics with no privacy policy is a trust problem and — for any EU/UK user — a GDPR problem.

Worth being explicit about in your privacy story regardless: with the Cohere provider, **document text leaves the machine.** The reranker (`reranker.ts:24`) posts chunk contents to Cohere, and the query rewriter posts the user's raw query. That's inherent to the design and fine — but it sits awkwardly against "local-first" and needs to be stated plainly.

**Fix:** build the Settings toggle (the backend is already done). Make telemetry opt-in, or show a first-run disclosure before the first event fires. Write a short privacy policy stating what leaves the device under each provider. Add a LICENSE.

## HIGH

### H1 — The app is not signed or notarized; it will not open for your users

`forge.config.ts:14-19` sets `osxSign: { identity: "-" }` — **ad-hoc signing**. An ad-hoc signed app cannot be notarized, and Gatekeeper quarantines any non-notarized app downloaded from the internet. Your users will double-click the DMG and get _"Commons is damaged and can't be opened. You should move it to the Trash."_ — which reads as malware, not as a config issue. There is no `osxNotarize` block at all. The Squirrel maker (line 51) has no `certificateFile`, so Windows shows a SmartScreen warning.

This is the hardest blocker: everything else in this report is invisible to a user who cannot launch the app.

**Fix:** Apple Developer Program ($99/yr), sign with a real _Developer ID Application_ cert, add `osxNotarize`. For Windows, a code-signing cert (Azure Trusted Signing is the cheap route).

### H2 — No auto-updater

Nothing references `autoUpdater` or `update-electron-app`. Combined with C2 and H1, every bug you ship is permanent for every user who installs it — you have no ability to push a security fix. `update-electron-app` against a public GitHub release feed takes about ten minutes and is far easier to add now than to retrofit onto an installed base.

### H3 — A failed sync flashes its error for 1.5 seconds, then says "Done"

`SyncPanel.tsx:71-75` — the auto-dismiss effect fires for **any** non-`syncing` status, including `"error"`:

```ts
if (status === "syncing") return;
const timer = setTimeout(onComplete, 1500); // also runs for status === "error"
```

So: expired Notion token, Cohere 401, network drop → the panel shows the error for 1500ms, then `onComplete()` unmounts the whole thing. The source silently displays its old document count. **The error state is never persisted anywhere**, so the user has no way to find out why their sync failed — or that it failed at all. And `SyncPanel.tsx:150` prints the literal string **`"Done"`** for `status === "error"`.

This is how C2's data loss stays invisible in practice.

**Fix:** only auto-dismiss on `complete`. Persist sync errors per-source and surface them in `SourceList`.

### H4 — "Clear all data" silently re-enables telemetry

`clearAllData` (`database.ts:473-480`) does `DELETE FROM settings`, dropping both `telemetry_enabled` and `device_id`. `handlers.ts:281` then immediately calls `initTelemetry(db)`, which reads the now-missing setting:

```ts
enabled = getSetting(db, "telemetry_enabled") !== "false"; // null !== "false" → true
```

Confirmed by test (`getSetting` returns `null` after the clear). A user who opted out is silently opted _back in_, with a fresh `device_id`. Given that "Clear all data" is exactly the button a privacy-conscious user reaches for, this is the worst possible place for this bug. (Today it's academic only because C3 means they could never opt out in the first place.)

### H5 — Deleted documents are never removed from the index

There is no reconciliation or tombstoning anywhere in the sync path — `syncSource` only ever inserts and updates. Delete a Notion page, or move a file out of the synced Drive folder, and its chunks remain in the database forever. They keep surfacing in search, and "Open source" leads to a 404. If a document was removed _because_ it shouldn't be accessible anymore, it stays searchable in Commons indefinitely.

**Fix:** have the connector report the full set of external IDs seen during a complete walk, then delete documents for that source not in the set. Guard it so a partial or aborted sync can never trigger a mass delete.

### H6 — Vector search silently truncates at 10,000 chunks

`getChunksWithEmbeddingsByModel` (`database.ts:321-332`) is `SELECT * FROM chunks … LIMIT 10000` with **no `ORDER BY`**. Past 10,000 chunks SQLite returns an arbitrary subset, and the rest of the corpus is simply absent from vector search — no error, no user-visible warning. Documents just quietly stop being findable. The `console.warn` at `searcher.ts:95` fires at 5,000 but goes to a log no user will ever read. 10,000 chunks is roughly 500-1,000 documents — well within what an active club accumulates.

For what it's worth, **speed is not the problem here.** I benchmarked it: reading 10k × 1536-dim embeddings and scoring them costs ~60ms of main-thread time, which is nothing next to a 200-500ms Cohere round trip. The _cap_ is the problem, not the CPU. Don't prematurely reach for a vector index — just raise or remove the limit (a full scan is fine to ~50k chunks) and warn the user if you ever do truncate.

### H7 — The renderer's sync state desyncs from the main process; background syncs are invisible

Main is the source of truth for "is this source syncing" (`activeSyncs`, `sync-handlers.ts:14`), but `SourceList.tsx:19` keeps its own `syncingIds` set and never asks main. There is no `getActiveSyncs()` in the preload surface. Consequences:

- The auto-sync scheduler broadcasts `sync:progress` to the window, but no `SyncPanel` is mounted to receive it — **background syncs are completely invisible**, and document counts silently change under the user.
- If the user clicks Sync on a source the scheduler is already syncing, `sync:start` throws and the UI shows the raw internal string `Sync already in progress for source: <uuid>` — then hides it after 1.5s per H3.

### H8 — Notion pages cannot be selected with a keyboard — the primary onboarding flow is blocked

`NotionPicker.tsx:138-142` — the row is a bare `<div onClick>` with no `role`, no `tabIndex`, no `onKeyDown`. The `Checkbox` inside is rendered **without** an `onChange` prop, so per `ui/checkbox.tsx:15-21` it degrades to a non-focusable `<span role="checkbox">`. A keyboard or screen-reader user literally cannot select any Notion page during setup. (`DrivePicker.tsx:172` does pass `onChange` and gets a real `<button>` — so the fix pattern already exists in your codebase.)

### H9 — ErrorBoundary doesn't cover onboarding or the shell, and "Try again" is a no-op loop

`App.tsx:166-189` wraps only the three page bodies. `OnboardingWizard` (line 123), the sidebar, `DragRegion`, and the initial spinner are all outside any boundary, and `main.tsx` has none around `<App />`. A throw in onboarding produces a blank white window with no recovery path.

Separately, `ErrorBoundary.tsx:32` — "Try again" only does `setState({ error: null })` and re-renders the _identical_ children with _identical_ props. For any deterministic render error it re-throws immediately. The button is a loop. There is no reset key, no remount, and no error reporting.

### H10 — Plaintext secrets are pulled into the renderer just to test whether they exist

`App.tsx:79` and `SettingsPage.tsx:56` call `loadSecret("cohere_api_key")` — which returns the **plaintext key** across the context bridge — when the only thing either site does with it is `key !== null`. `hasSecret()` already exists in the preload (`index.ts:73-74`) and is used correctly for Notion/Drive at `SettingsPage.tsx:59-60`. Same pattern at `ConnectNotionButton.tsx:18` and `ConnectDriveButton.tsx:18`, which hoist raw OAuth tokens into renderer memory.

This needlessly exposes credentials to renderer-side XSS, devtools inspection, and crash dumps. Free fix: use `hasSecret()`.

### H11 — The embedding-mismatch warning never fires for the flow that causes the mismatch

`SearchPage.tsx:30-37` gates `checkEmbeddingHealth()` behind `healthLoaded.current`, so it runs **once per app session**. But the mismatch is _created at runtime_ by `SettingsPage.tsx:124-136`, which switches provider without re-embedding. So the exact flow that produces mismatched chunks — "switch Cohere → Ollama with 5,000 chunks, confirm" — never surfaces the warning. The user searches, gets results scored against a different vector space, and sees no explanation until they restart the app. Health is also never re-checked after a sync.

### H12 — A provider switch can leave the app silently broken

`SettingsPage.tsx:128-132` — `onProviderReset()` is only called when switching _to_ Ollama. Going **Ollama → Cohere with no stored key** sets the provider and returns; `App`'s `ready` flag is never re-evaluated, so the app stays in the main UI and every subsequent search fails at the embedder.

Symmetrically, switching **to Ollama** calls `onProviderReset()`, which sets `ready = true` **without ever checking that Ollama is installed or running** (`checkOllama` is never called from Settings, and `App.tsx:75-77` short-circuits on `provider === 'ollama'`). The user can trivially configure a provider that does not exist, skip onboarding entirely, and get raw `fetch` errors on every search.

## LOW

- **Onboarding has no persistence.** `step` is local state only. Validate the API key (which writes the secret immediately, `ApiKeyForm.tsx:19-20`) then quit — the next launch finds the key, sets `ready = true`, and drops the user straight onto Search. They never see "Connect your docs," and `SearchPage` has no "no sources connected" empty state distinct from "no results." _(`OnboardingWizard.tsx:30`)_
- **Non-Latin documents chunk badly.** The sentence splitter (`chunker.ts:59`) requires an `[A-ZÀ-ɏ"]` lookahead, so CJK/Arabic/Hebrew never splits into sentences; it falls through to whitespace word-splitting, which CJK doesn't have, producing one enormous chunk. The FTS tokenizer (`porter unicode61`) doesn't segment CJK either.
- **No `crashReporter`, no `uncaughtException`/`unhandledRejection` handler** in main. A main-process crash is a silent death with zero diagnostics — and you won't know it happened.
- **`shutdownTelemetry()` is not awaited** on `will-quit`; buffered events are dropped on exit.
- **The 5-minute OAuth timeout timer is never cleared** on success, holding a handle for 5 minutes after the flow completes. _(`notion-oauth.ts:179`, `google-oauth.ts:161`)_
- **10 dependency vulnerabilities (7 high)** in `tar`/`tmp`/`esbuild` — but **`pnpm audit --prod` is clean.** They all arrive via `electron-forge` and `vite`, so nothing vulnerable ships. Update anyway.
- **Entitlements are maximally permissive** — `disable-library-validation`, `allow-dyld-environment-variables`, and `allow-unsigned-executable-memory` together undo most of the hardened runtime. Trim to what `better-sqlite3` actually needs.
- **CSP could be tighter**: no `object-src 'none'`, `base-uri 'self'`, or `form-action 'none'`. The renderer makes no direct network calls at all (everything goes through IPC), so `connect-src 'none'` would be the accurate policy.
- **`package.json` scaffold placeholders**: `author: "example.com"`, `homepage: "https://electron-vite.org"`, no `license`, `version: 0.1.0`. These appear in the packaged app's metadata.
- **Accessibility gaps**: no `aria-current` on nav (`App.tsx:139-151`), search results not in an `aria-live` region so screen-reader users get no announcement that a search completed (`SearchPage.tsx:95-121`), no focus management after search.
- **`formatRelativeTime` renders "NaNm ago"** on an unparseable timestamp (`SettingsPage.tsx:21-29`).
- **`SettingsPage.tsx:126` skips the provider-switch confirm entirely when `stats` is `null`** — so if the stats fetch failed, a switch that invalidates every chunk happens with no prompt.
- **Dead code**: `DriveFolderInput.tsx` (68 lines) is imported nowhere.
- **`.DS_Store` is tracked in git** (3 files).

## What's already right

Worth stating explicitly, because a lot of this is done properly and shouldn't be touched:

- **Electron security baseline is correct**: `contextIsolation: true`, `sandbox: true`, `nodeIntegration: false`, a real CSP, `setWindowOpenHandler` denying by default, and `openExternal` validated against an http/https allowlist — correctly blocking `javascript:` and `file:`.
- **Zero XSS surface**: no `dangerouslySetInnerHTML`, no `innerHTML`. All Notion/Drive content renders as escaped JSX text children. Untrusted URLs never reach an `href`. Notion icons are narrowed to emoji-only at the connector.
- **Secrets are handled well**: OS keychain via `safeStorage`, a dedicated `secrets` table, an allowlist on key names at the IPC boundary, and graceful handling of corrupt ciphertext.
- **OAuth is implemented correctly**: PKCE with S256, `state` validated against CSRF, loopback bound to `127.0.0.1`, single-flight guarding, and a timeout.
- **`SearchPage` correctly guards against out-of-order responses** with a monotonic `requestIdRef` (`SearchPage.tsx:49-62`). This is the classic bug and it isn't there.
- **All SQL is parameterized.** No injection anywhere.
- **Hybrid retrieval — FTS5 + vector, merged with RRF, then Cohere rerank — is a genuinely good design**, better than most RAG apps at this stage.
- **151 tests pass; typecheck is clean.** `OllamaOption` and `SettingsPage` both handle effect cleanup correctly; `ResultCard` is properly memoized with stable keys.

## Recommended order

**Before you ship anything**

1. **C1** — rotate the OAuth secrets, move token exchange behind a backend.
2. **C2** — fix the content-hash-before-embedding bug. Add the regression test.
3. **C3** — build the telemetry toggle (backend is already done), add a privacy policy and LICENSE.
4. **H1** — real code signing + notarization.
5. **H2** — add an auto-updater.

**Before you call it 1.0** 6. **H3, H4** — sync failures must persist and be visible; the opt-out must survive a data clear. 7. **H5, H6** — reconcile deleted documents; stop silently truncating the corpus at 10k chunks. 8. **H7–H12** — sync-state desync, keyboard access in the Notion picker, ErrorBoundary coverage, `hasSecret()` instead of `loadSecret()`, and the two provider-switch dead ends. 9. **M1–M3** — make the database resilience code actually work. Right now it's decorative.

**Then:** the rest of MEDIUM, then LOW.

The foundation is good. What's broken is the shipping story and the error paths — and error paths are exactly where a "local-first, trust me with your club's documents" app has to be strongest.
